01 Đánh giá thẳng thắn về "Microservice"

Sự thật về microservice cho blog cá nhân

Microservice thật (mỗi service một database, giao tiếp qua network, deploy độc lập, service mesh…) sinh ra để giải quyết vấn đề tổ chức nhiều team và scale không đồng đều. Blog một người dùng không có hai vấn đề đó.

Nếu bê nguyên microservice lên một VPS 4GB RAM:

  • ✗ Mỗi Go service ~30–80MB, nhưng Postgres + Redis + Prometheus + Grafana + Loki mới là thứ ngốn RAM (1.5–2.5GB).
  • ✗ Debug khó gấp bội: một request đi qua 3 service = 3 nơi có thể lỗi.
  • ✗ Transaction phân tán, eventual consistency — bài toán khó không cần thiết.

Giải pháp đúng: Modular Monolith + 1 service tách riêng

Kiến trúc học được đủ điều hay của microservice mà không trả giá vận hành:

┌─────────────────────────────────────────────────────────┐
│  API Core (Go monolith, chia module rõ ràng)            │
│  [ auth ] [ posts ] [ media ] [ chat/rag ]              │
│  Mỗi module = 1 package Go, interface rõ ràng            │
└────────────────────────┬────────────────────────────────┘
                         │ gRPC / message queue (NATS / Redis Stream)
                         ▼
┌─────────────────────────────────────────────────────────┐
│  AI Worker (service RIÊNG — Python, LangChain/LangGraph)│
│  - Research agent · sinh embeddings · gọi LLM · viết draft
│  - Chạy cron job 09:00                                    │
└─────────────────────────────────────────────────────────┘

Vì sao tách AI worker là hợp lý (không over-engineer): hệ sinh thái AI mạnh nhất ở Python (bạn học Go cho core, Python cho AI — đúng thực tế doanh nghiệp); tải research nặng/lâu nên tách để không block API; deploy độc lập; và bạn vẫn học thật inter-service, container, health check.

Khuyến nghịCore = Go monolith module hoá. AI worker = Python (LangChain). Đây là mô hình "Hybrid" tốt nhất và trùng đúng việc bạn đang học LangChain.

03 Phần AI — Phân tích chi tiết

3.1 · Chatbot RAG hỏi về website

Khi đăng/sửa bài:  post → chunk → embedding → lưu pgvector
Khi user hỏi:      câu hỏi → embedding → similarity search (pgvector)
                   → top-k chunk liên quan → prompt LLM → trả lời kèm trích dẫn

Câu hỏi kiểu "web có bài nào về X" nên xử lý hybrid: intent đơn giản → query metadata (không cần LLM); câu hỏi mở → RAG. Model embedding rẻ: text-embedding-3-small, Google text-embedding-004 (free tier rộng), hoặc chạy local bge-small (0đ).

3.2 · AI research + tự viết + Telegram 9h sáng (human-in-the-loop)

Tính năng "signature" của dự án:

09:00 (cron trong AI worker):
  1. Lấy danh sách chủ đề quan tâm
  2. Web search  → Tavily (free tier) / Brave Search
  3. Fetch & đọc nguồn → tóm tắt, trích dẫn
  4. LLM viết draft + tiêu đề + tags + meta SEO
  5. Lưu draft (status = PENDING_APPROVAL)
  6. Gửi Telegram: tiêu đề + tóm tắt + [✅ Duyệt] [✏️ Sửa] [❌ Bỏ]
  7. Bạn bấm nút → publish / xoá / mở editor chỉnh

Framework: LangGraph (thuộc LangChain) — hợp luồng nhiều bước có state + human-in-the-loop. Telegram dùng inline keyboard, webhook về AI worker.

Ranh giới bắt buộcAI KHÔNG BAO GIỜ tự đăng khi chưa có approve của bạn — tránh hallucination lên blog công khai.

3.3 · Trợ lý viết trong editor

Streaming completion: bôi đen text → "viết tiếp / tóm tắt / đổi văn phong / dịch". Nếu chọn editor Novel thì có sẵn slash-command AI.

04 Observability / Monitoring (Grafana)

Full stack LGTM (Loki + Grafana + Tempo + Prometheus) rất "ngon" để học nhưng ngốn 1.5–2.5GB RAM — cạnh tranh trực tiếp với app trên VPS rẻ.

Lựa chọnNội dungKhi nào
A · Self-host gọnPrometheus + Grafana + Loki (+ Tempo sau). OpenTelemetry instrument Go + Python.VPS ≥ 4GB. Học đầy đủ metrics/logs/traces.
B · Grafana Cloud Free Khuyến nghịFree tier: 10k series, 50GB logs, 50GB traces/tháng. VPS không tốn RAM cho monitoring.VPS ≤ 2GB hoặc muốn tiết kiệm RAM.
Khuyến nghịBắt đầu với B (Grafana Cloud Free) để dồn RAM cho app, rồi thử self-host A trong một phase riêng để học vận hành. Metrics tối thiểu nên có: RED (rate/error/duration), token LLM + chi phí ước tính, thời gian job research, số bài publish.

  1. Chọn model theo việc: research/viết dùng model rẻ; model mạnh chỉ khi thật cần.

12 craft.js, Reka.js & Visual Page Builder

12.1 · craft.js là gì (chính xác)

craft.js không phải rich text editor, cũng không phải page builder sẵn dùng. Nó là framework/toolkit React để BẠN TỰ XÂY một page builder: drag & drop component vào canvas, node tree, settings panel, serialize ra JSON. Bạn tự dựng UI của builder → đổi lại là tự do tuyệt đối.

Phân biệtRich text editor (Tiptap) lo nội dung chữ trong 1 bài; page builder (craft.js/Puck) lo bố cục cả trang bằng kéo-thả block/component. Hai thứ bổ sung, không thay thế nhau.

12.2 · Vấn đề của craft.js (kiểm chứng 2026-07)

  • Vẫn hoạt động nhưng bảo trì chậm lại rõ rệt.
  • Lý do: tác giả (Prev Wong) dồn sức cho dự án kế thừa Reka.js.
  • Hệ quả: hỗ trợ React 19 còn lấn cấn; rủi ro production lâu dài cao hơn lib đang active.
Khuyến nghị thay thếDùng Puck — visual editor cho React, đang bảo trì tích cực (~13K sao), hợp Next.js App Router, có UI sẵn dùng ngay (craft.js phải tự dựng nhiều hơn).

12.3 · Reka.js là gì & khác craft.js

Reka.js = hệ thống quản lý state để xây trình no-code/page builder (cùng tác giả). Dựa trên một AST, cho phép người dùng cuối tạo component có logic động (props, state, if, loop) — như "mini-React" trong builder.

craft.jsReka.js
Vai tròFramework dựng UI của builderEngine state bên dưới
Người dùng cuốiKéo-thả component tĩnhTạo component có logic động
OutputJSON node-treeView tree JSON framework-agnostic
Cộng thêm—Collaboration realtime (CRDT/Yjs)
Trạng tháiBảo trì chậmMới, mạnh hơn nhưng phức tạp & ngách

12.4 · Ứng dụng "hay ho" cho blog

  1. Trình dựng trang chủ / landing / about tuỳ biến (lưu layout JSON).
  2. Custom layout cho từng bài đặc biệt (showcase) — vượt giới hạn rich text.
  3. 🌟 AI sinh layout: vì output là JSON, cho LLM sinh JSON layout. "AI, tạo landing cho series bài Go" → AI trả JSON → builder render. Kết nối đẹp giữa AI và page builder.
  4. Trình tạo template newsletter/email kéo-thả.
  5. Widget/section builder cho trang chủ/sidebar.

12.5 · Lời khuyên (chốt)

Đáng làm, nhưng để Phase 5 — sau khi blog + Tiptap + AI cơ bản đã chạy. Thứ tự lựa chọn giảm dần:

  1. Puck — bắt đầu bằng cái này (active, có UI sẵn, hợp Next.js).
  2. craft.js — chỉ khi cần tuỳ biến UX builder sâu hơn Puck (chấp nhận rủi ro bảo trì).
  3. Reka.js — chỉ khi tham vọng làm nền tảng no-code động thật sự (vượt nhu cầu blog).
Không làm sớmĐừng đưa page builder vào giai đoạn đầu. Ưu tiên rich text (Tiptap) cho việc viết bài trước. Điểm chung cả ba: đều serialize ra JSON → giữ được ý tưởng "AI sinh layout".

14.3 · Điểm cần lưu ý khi làm admin SPA

  • Auth: Google OAuth theo BFF pattern (đã chốt) — SPA không tự giữ token, chỉ gọi API kèm credentials: 'include' + bọc protected route. Chi tiết ở 14.4.
  • Routing: TanStack Router file-based (src/routes/, routeTree.gen.ts sinh tự động), search params type-safe + route loaders tích hợp TanStack Query.
  • Bảng dữ liệu: TanStack Table (headless) qua DataTable<TData> dùng chung — sort server-side theo URL search, ẩn/hiện cột.
  • shadcn/ui: hỗ trợ Vite tốt — cấu hình components.json trỏ về packages/ui để dùng chung với web.
  • Editor Tiptap và luồng presigned URL upload R2 nằm trong app admin.
  • Deploy: build static → Cloudflare Pages / Vercel (static) hoặc phục vụ qua Coolify. Không cần Node runtime SSR.

14.4 · Luồng Google OAuth (BFF pattern)

Nguyên tắcSPA không bao giờ chạm vào token. Go core đóng vai Backend-for-Frontend (BFF): là OAuth client, đổi code, chỉ trả về trình duyệt một session cookie httpOnly. Đây là khuyến nghị chính thức trong OAuth 2.0 for Browser-Based Apps (BCP) — an toàn hơn hẳn SPA tự giữ JWT (miễn nhiễm XSS đánh cắp token).
1. User bấm "Đăng nhập với Google"  → SPA điều hướng: GET {core}/auth/google/login
2. Go core tạo state + PKCE, redirect sang trang consent của Google
3. User đồng ý → Google redirect: GET {core}/auth/google/callback?code=...&state=...
4. Go core:
     - kiểm tra state (chống CSRF)
     - đổi code → token với Google (client_secret + code_verifier)
     - đọc email + email_verified từ ID token
     - ✅ KIỂM TRA ALLOWLIST: email có thuộc danh sách admin? không thì 403
     - tạo session (Redis/DB), set cookie:
         Set-Cookie: sid=<...>; HttpOnly; Secure; SameSite=Lax; Path=/
5. Go core redirect về admin SPA (đã đăng nhập)
6. Mọi request sau: SPA gọi API với credentials:'include' → cookie tự đính kèm
7. Logout: POST {core}/auth/logout → xoá session + clear cookie
Điểm mấu chốtAllowlist email là bắt buộc — nếu không, bất kỳ ai có tài khoản Google đều login được. client_secret chỉ nằm ở Go core (lý do phải dùng BFF). Nếu admin & core khác origin → cần CORS Allow-Credentials: true + SameSite=None; Secure. Thư viện Go: golang.org/x/oauth2 + oauth2/google, tự thêm PKCE + state.

16 TypeScript nâng cao

Đoạn type nội bộ của react-hook-form dùng conditional/mapped/template-literal/recursive types. Hoàn toàn nên viết ở tầm đó và rất đáng học. Áp dụng cụ thể:

// 1) Branded types — ID an toàn kiểu
type Brand<T, B> = T & { readonly __brand: B };
type PostId = Brand<string, 'PostId'>;
type UserId = Brand<string, 'UserId'>;

// 2) Zod = single source of truth
const PostSchema = z.object({
  id: z.string(),
  title: z.string().min(1),
  status: z.enum(['DRAFT', 'PENDING_APPROVAL', 'PUBLISHED']),
});
type Post = z.infer<typeof PostSchema>;   // type tự sinh, luôn khớp validation

// 3) Recursive/mapped types (giống DeepRequired)
type DeepReadonly<T> = T extends BrowserNativeObject
  ? T : { readonly [K in keyof T]: DeepReadonly<T[K]> };

// 4) Template literal types
type Permission = `post:${'read' | 'write' | 'delete'}`;

// 5) Discriminated unions cho state AI draft
type DraftState =
  | { status: 'researching' }
  | { status: 'ready'; draft: Post; sources: string[] }
  | { status: 'failed'; error: string };

Thư viện hỗ trợ

  • Zod — validation + suy ra type. Nền tảng mọi thứ.
  • type-fest — kho utility type nâng cao (đúng loại trong đoạn code bạn dán).
  • ts-reset — vá type built-in "lỏng" của TS thành chặt hơn.
Vũ khí quan trọng nhấtEnd-to-end type safety Go ↔ TypeScript. Vì BE là Go (không dùng tRPC), dùng contract-first codegen: (A) REST + OpenAPI (swaggo → openapi-typescript/orval sinh client + TanStack hooks), hoặc (B) gRPC + Protobuf (buf + Connect sinh type Go và TS từ cùng file .proto). Đổi field trong Go → FE compile lỗi ngay.
